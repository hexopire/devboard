● Task 1.3 done. 7 migration files in backend/migrations/, numbered 001–007 in
  dependency order (users → teams → team_members → projects → tasks → comments →
  attachments).

  Key decisions, and why:
  - Used native Postgres ENUM types (user_role, team_member_role, task_status)
  over CHECK constraints — Postgres stores an enum as a 4-byte value, and it's
  self-documenting in \d tablename. Trade-off: adding a new value later needs
  ALTER TYPE ... ADD VALUE, a real migration, not just an app-level constant
  change.
  - team_members uses a composite primary key (team_id, user_id) instead of a
  surrogate id — it's the natural key and already enforces "one membership per
  user per team," so no separate UNIQUE needed.
  - Cascade direction matters per relationship:
  projects/tasks/comments/attachments all cascade-delete when their parent
  (team/project/task) is deleted — deleting a team should wipe everything under
  it. But tasks.assignee_id uses ON DELETE SET NULL, not cascade — deleting the
  assignee should orphan the assignment, not destroy the task itself.
